Project Overview

LibraFlow is a full-stack Library Management System built for academic institutions. It allows students to search, reserve, and borrow books, while librarians manage the full inventory, loans, and reports through a dedicated admin panel.

Setting Up Your Development Environment

Prerequisites

  • Python 3.10 or higher installed
  • Git installed and configured with your GitHub account
  • A code editor (VS Code recommended)
  • Terminal / Command Prompt access

Step 1 — Clone the Repository

git clone https://github.com/Aryankanani1/LibraFlow.git
cd LibraFlow

Step 2 — Create a Virtual Environment

# macOS / Linux
python3 -m venv venv
source venv/bin/activate

# Windows
python -m venv venv
venv\Scripts\activate
Note
You should always see (venv) at the start of your terminal prompt before running any commands.

Step 3 — Install Dependencies

pip install -r requirements.txt

Step 4 — Initialize the Database

python app.py

This creates libraflow.db in the project root and seeds 2 demo users on first run.

Step 5 — (Optional) Seed 63 Real Books

python seed_books.py

Step 6 — Run the Application

python app.py

Open your browser and go to: http://127.0.0.1:5000

macOS SSL Fix (Python 3.13)
If you get an SSL certificate error when searching for books, run:
/Applications/Python\ 3.13/Install\ Certificates.command

Git Workflow — Branching Strategy

We follow a feature branch workflow. The main branch is protected — nobody pushes directly to it.

Branch Naming Convention

TypePatternExample
New featurefeature/descriptionfeature/export-reports
Bug fixfix/descriptionfix/loan-due-date-bug
UI improvementui/descriptionui/mobile-navbar
Teststest/descriptiontest/auth-routes
Documentationdocs/descriptiondocs/api-endpoints

Branch Rules

  • main — production-ready code only. Protected. Requires PR + 1 approval to merge.
  • feature/* — one feature per branch. Delete after merge.
  • Never commit directly to main.
  • Keep branches short-lived. Merge within a few days of starting.
  • Pull from main before starting a new branch to stay up to date.
Rule
Do not work on main directly. If you accidentally commit to main, stop immediately and ask the project lead before doing anything.

How to Make Changes (Step-by-Step)

Follow these steps every time you work on something — no exceptions.

1
Sync with main before starting 
git checkout main
git pull origin main
2
Create a new branch for your work 
git checkout -b feature/your-feature-name
3
Make your changes

Edit files in your code editor. Test locally using python app.py.

4
Stage and commit your changes 
git add filename.py         # Add specific files (preferred)
git add .                   # Or add all changes
git commit -m "feat: add export to CSV for reports"
5
Push your branch to GitHub 
git push origin feature/your-feature-name
6
Open a Pull Request on GitHub

Go to github.com/Aryankanani1/LibraFlow → Click Compare & pull request → Fill in title and description → Submit.

7
Wait for review and approval

The project lead reviews your PR. If changes are requested, push new commits to the same branch — the PR updates automatically.

8
After merge — clean up your branch 
git checkout main
git pull origin main
git branch -d feature/your-feature-name

Commit Message Rules

We follow the Conventional Commits standard. Every commit message must follow this format:

<type>: <short description in lowercase>

Types

TypeWhen to useExample
featNew feature addedfeat: add CSV export for reports
fixBug fixfix: correct overdue fine calculation
uiVisual/style change onlyui: improve mobile navbar layout
refactorCode cleanup, no behavior changerefactor: extract loan helper function
testAdding or updating teststest: add auth route unit tests
docsDocumentation onlydocs: update setup instructions
choreConfig, dependencies, miscchore: update requirements.txt

Rules

  • Keep the subject line under 72 characters
  • Use lowercase, no period at the end
  • Be specific — say what changed, not just "update"
  • One logical change per commit — do not mix unrelated changes
Bad Examples — Do Not Do This
git commit -m "changes"
git commit -m "update stuff"
git commit -m "WIP"
git commit -m "fixed"
Good Examples
git commit -m "feat: add barcode scanner to issue book form"
git commit -m "fix: student cannot reserve already-issued book"
git commit -m "ui: add loading spinner to book search"

Pull Request Rules

PR Title

Same format as commit messages: feat: add CSV export for reports

PR Description Must Include

  • What changed — brief description of what you built or fixed
  • Why — reason for the change
  • How to test — steps to verify the change works
  • Screenshots — if it's a UI change, attach before/after screenshots

PR Template (copy-paste this)

## What changed
- Brief description of the change

## Why
- Reason for this change

## How to test
1. Run the app: python app.py
2. Log in as admin / student
3. Navigate to [page]
4. Verify [expected behavior]

## Screenshots (if UI change)
[Attach screenshot here]

Rules for Reviewers

  • Review PRs within 24 hours of submission
  • Test the change locally before approving
  • Leave specific comments — not just "looks good"
  • Do not merge your own PR — someone else must approve

Rules for PR Authors

  • Keep PRs small and focused — one feature or fix per PR
  • Do not force-push to a branch that has an open PR
  • Respond to review comments within 24 hours
  • Resolve all comments before asking for re-review
Merge Strategy
We use Squash and Merge to keep the main branch history clean. This combines all your branch commits into one clean commit on main.
09
Code Standards

Python

  • Follow PEP 8 — 4-space indentation, snake_case for variables/functions
  • Route functions named clearly: issue_book(), not ib()
  • All routes must have @login_required where appropriate
  • Admin-only routes must check current_user.role == 'librarian'
  • Do not hardcode values — use config or constants
  • No print() statements in committed code — use Flask's logger

HTML / Jinja2 Templates

  • All pages must {% extends 'base.html' %}
  • Use {% block title %} and {% block content %}
  • Use Lucide icons: <i data-lucide="icon-name"></i> — no Font Awesome
  • Do not put <style> blocks in templates unless print-specific (like barcode page)
  • Use existing CSS classes from style.css — do not inline styles unless necessary

CSS

  • All styles go in static/css/style.css
  • Use CSS variables: var(--primary), var(--muted), etc. — do not hardcode hex values
  • Font: Roboto Condensed only — do not introduce new fonts
  • Colors: indigo accent only (#4f46e5). Use neutral grays for everything else
  • Do not add new color accents — keep the palette minimal

Database

  • Never delete libraflow.db in a shared environment without team notice
  • All model changes must be reflected in models.py
  • If you add a new column, update the seed data accordingly
  • Do not commit libraflow.db to git
CSS Variables Reference
--primary: #4f46e5  |  --primary-dark: #3730a3  |  --muted: #64748b
--success: #16a34a  |  --danger: #dc2626  |  --warning: #b45309

Rules & Regulations
Strictly Prohibited
  • Pushing directly to main
  • Merging your own Pull Request
  • Committing libraflow.db, .env, or any credential files
  • Using git push --force on shared branches
  • Deleting another team member's branch without permission
  • Introducing new external dependencies without team discussion
Required Before Every PR
  • Test the app locally — make sure nothing is broken
  • Run python -m pytest if tests exist
  • Check that your code follows the standards in Section 9
  • Make sure the PR description is complete (Section 8)
Best Practices
  • Keep your local main always up to date with git pull
  • Commit often with small, logical commits — do not build for 3 days and commit everything at once
  • If a feature is large, break it into smaller PRs
  • Write a comment in the code only when the logic is not obvious
  • Delete your branch after it is merged

What Happens if Rules are Broken

ViolationAction
Direct push to mainRevert commit, team discussion
Committed credentials or DB fileImmediately rotate secrets, force-push revert
Broken PR merged to mainRevert merge, fix in new branch, re-review

Running Tests

Install Test Dependencies

pip install pytest

Run All Tests

python -m pytest tests/ -v

Run a Specific Test File

python -m pytest tests/test_auth.py -v

Test Files

FileTests
tests/test_auth.pyLogin, logout, registration, role checks
tests/test_models.pyUser, Book, Loan, Reservation, Review models
tests/test_student.pyDashboard, search, borrow, reserve, review
tests/test_admin.pyAdmin panel, issue/return, reports
Note
Tests use an in-memory SQLite database — they do not affect your local libraflow.db. Always run tests before opening a Pull Request.

Deployment Notes

LibraFlow is currently a development project. These are notes for when deployment is considered.

Before Deploying

  • Set DEBUG = False in config.py
  • Change SECRET_KEY to a strong random value — never use the dev key
  • Switch from SQLite to PostgreSQL or MySQL for production
  • Set up proper environment variables — never hardcode secrets in code
  • Run all tests and ensure they pass

Environment Variables (Production)

SECRET_KEY=your-random-secret-key-here
DATABASE_URL=postgresql://user:pass@host/dbname
FLASK_ENV=production
